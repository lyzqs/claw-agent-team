# 第二轮 Grafana 人类可读性改造与缺失指标清单（Issue #32 PM）

## 1. 文档目标

本文用于定义 `agent-team-grafana` 项目的第二轮 Grafana 迭代清单与人类可读性改造规格，回答以下问题：

1. 当前已落地看板中，哪些面板虽然“有数据”，但不够直观、不够好读，应该调整表达方式。
2. 哪些面板仍然容易出现 `No Data` 或空洞呈现，应该修复、替换或删除。
3. 哪些底层指标名称或分组方式对普通用户不友好，应该收口为更人类可读的视角。
4. OpenClaw、NewAPI、Agent Team 等系统当前还缺哪些关键指标，才能更贴近用户实际关心的问题。
5. 第二轮迭代应如何拆成可独立验收的实现 issue，避免再次把多个目标混成一个大包。

本文是 `Issue #32` 的直接交付物，供 `Issue #33` 到 `Issue #36` 实现使用。

## 2. 事实基础与当前已知线索

基于当前仓库中的 Grafana 蓝图、各系统 PM 规格文档与已落地 dashboard JSON，已确认以下事实：

1. 当前本地 Grafana 已经不是空白底座，而是已经有一套跨系统 dashboard 交付，至少包括：
   - 主机系统
   - Agent Team
   - NewAPI
   - Arena
   - Uptime Kuma
   - OpenClaw

2. 当前 dashboard 设计整体遵循第一轮蓝图，但大量页面仍保留较多 `stat + bargauge` 组合，趋势表达与人类解释层不足。
   - 例如 OpenClaw 三套看板中仍有大量 `bargauge` 面板。
   - Agent Team 运行总览、流转健康等看板也仍保留多个分布型 `bargauge` 面板。
   - NewAPI 概览页虽然已有趋势图，但渠道 / 模型维度仍偏“指标开发者视角”，不是“业务使用者视角”。

3. 从已落地 dashboard 观察，当前最明显的可读性问题主要集中在以下几类：
   - **底层实现维度直接暴露给用户**，例如 `lane`、`outcome`、`completion_mode` 等。
   - **分布型面板过多采用 bargauge**，导致“看起来有值，但不方便理解变化趋势”。
   - **部分标题虽然中文化了，但仍然更像工程字段翻译，而不是自然问题表达。**
   - **空数据面板缺少语义兜底**，用户无法区分“当前无事件”与“当前接入缺失/面板坏了”。

4. 第一轮 PM 规格已分别定义了各系统“该看什么”，但第二轮用户诉求已经从“有没有指标”提升为“是不是直观、易读、贴近真实管理问题”。
   - 这意味着第二轮重点不再是简单补指标，而是重新审视哪些指标该直接给人看，哪些应下沉到 drill-down。

5. 当前用户明确提出的新增关键视角至少包括：
   - OpenClaw：按 `agent` 维度看消息量与 token 变化趋势。
   - NewAPI：按 `channel` 维度看 token 变化趋势。
   - Agent Team：按 `project` 维度看 issue 新增与完成情况。

6. 历史 follow-up 已经完成过一轮迭代，但仍然存在剩余缺口：
   - `Issue #18` 已拆成中文化、No Data 修复、Arena 补齐、进程时点增强。
   - `Issue #19` 已拆成 Agent Team 吞吐量增强。
   - `Issue #25` 已拆成 OpenClaw 规格与实现。
   - 本轮 `Issue #31` 说明这些交付虽然成立，但还不足以满足“直观且人类易读”的第二轮目标。

## 3. 第二轮迭代的设计目标

第二轮不再追求“把所有指标都塞到看板上”，而是明确追求以下 4 个目标：

### 3.1 从工程可观测转向人类可读

第一轮重点是：
- 指标接出来
- 看板跑起来
- 基础运行态可看

第二轮重点是：
- 普通人能不能一眼看懂
- 是否能直接回答用户真正关心的问题
- 是否减少对底层术语和内部实现细节的依赖

### 3.2 从静态分布转向趋势优先

用户当前明确偏好：
- 更偏向 `timeseries`
- 少用只展示快照排序的 `bargauge`

因此第二轮默认策略是：
- 如果问题本质是“变化情况如何”，优先 timeseries。
- 如果问题本质是“当前 TopN 排名”，才保留 bargauge / table。

### 3.3 从“展示有值”转向“解释问题”

面板不只是要显示数字，而是要帮助用户回答：
- 今天比昨天更好吗
- 哪个维度在恶化
- 哪个系统值得看一眼
- 为什么现在没有数据或没有动作

### 3.4 从单看板优化转向跨系统统一体验

第二轮需要统一的不是单个 dashboard，而是跨项目的阅读体验：
- 相似的问题用相似的表达方式
- 相似的趋势用相似的 panel 类型
- 相似的空数据用相似的处理策略
- 相似的标题应能直接转成人类问题语言

## 4. 第二轮人类可读性基线

### 4.1 Timeseries-first 适用范围

以下场景默认优先 `timeseries`：

1. **变化趋势**
   - 消息量变化
   - token 变化
   - issue 新增/关闭变化
   - 请求量变化
   - 错误率变化
   - 队列等待时间变化

2. **多维度对比但核心仍是随时间变化**
   - 按渠道看 token 趋势
   - 按 agent 看消息量趋势
   - 按项目看 issue 新增/关闭趋势
   - 按 provider/model 看 token 趋势

3. **需要观察波动、异常峰值或持续恶化**
   - run duration
   - queue wait
   - webhook duration
   - request latency / error rate

### 4.2 Bargauge 保留范围

以下场景可以继续使用 `bargauge`，但应受控：

1. 当前时点 TopN 排名
   - 当前 Top 进程 CPU / 内存
   - 当前 Top 渠道错误率
   - 当前 backlog Top 项目

2. 短时间内无需强调趋势，只需要快速排序
   - 当前 down monitor 排名
   - 当前错误码分布前几位

3. 用户天然用“排行榜”理解的问题

### 4.3 应优先删除或下沉的表达

以下类型不应继续留在主视图：

1. **底层实现维度且普通用户难理解的面板**
   - `Queue Wait 按 Lane`
   - `Queue Depth 按 Lane`
   - 直接暴露 `completion_mode` 的总览主面板
   - 直接暴露 `outcome`、`state` 但无解释语义的主面板

2. **只能看排序、看不出变化趋势的核心业务面板**
   - 例如“token 分布”“message 分布”若是用户真正关心变化，主视图应改为 timeseries，分布图退居次级位置。

3. **空数据出现时无法判断原因的面板**
   - 没有数据兜底说明
   - 当前环境未启用该能力却直接渲染 `No Data`
   - 数据源未打通但界面看起来像坏了

## 5. No Data 面板处理策略

第二轮对 `No Data` 的处理必须统一，不允许继续把含混状态直接暴露给用户。

### 5.1 三类 No Data 的区分

#### A. 当前无事件，但链路是通的
例如：
- 24h 内没有 webhook error
- 当前没有 stuck session
- 当前没有 down monitor

处理策略：
- 不应显示误导性的空图。
- 应优先使用 `stat + 0`、空态文案或明确的“当前无异常事件”。

#### B. 当前环境未启用该能力
例如：
- 某环境没有 webhook 型 channel
- 某环境没有某类数据源

处理策略：
- 不应继续占主视图核心位置。
- 需要改为带说明的替代面板，或直接从该环境默认视图中移除。

#### C. 采集/实现链路未完成或已损坏
例如：
- exporter 没暴露
- query 失效
- label 已变但面板未同步

处理策略：
- 这属于实现缺陷，应该在第二轮整改中修复或删除，不能继续作为“可接受空态”存在。

### 5.2 主视图约束

任何位于第一行或第二行的核心面板，如果持续处于 B / C 类 `No Data`，必须：
1. 修复数据链路，或
2. 降级到次级视图，或
3. 直接移除

不能继续占据首页高价值位置。

## 6. 现有面板的第二轮问题清单

以下清单只记录当前最明显、最符合用户反馈的问题类型，供 Dev 实现时按清单逐项处理。

### 6.1 OpenClaw

#### 当前主要问题

1. 三套看板中 `bargauge` 数量偏多，开发者视角过重。
2. `Queue Depth 按 Lane`、`Queue Wait 按 Lane` 对普通用户难以理解。
3. `Provider / Model Token 分布`、`Provider / Model 成本分布` 只能看累计分布，不够直接回答“变化情况”。
4. 当前主视图更偏 provider / model / queue，而用户已经明确提出更想看 `agent` 维度的消息量与 token 变化。
5. `Webhook` 相关面板在某些环境可能长期空置，主视图易出现语义不明的空值或低价值面板。

#### 建议整改

- 主视图中减少 lane 维度的直接暴露，把“Queue Wait / Depth 按 Lane”降级为二级排障面板或改名后放入次级视图。
- `Provider / Model` 相关分布型 bargauge 保留为补充视图，但主视图应增加趋势面板。
- 重点新增 `agent` 维度消息量与 token 变化趋势。
- 对 webhook 相关面板加环境适配策略，避免主视图长时间空置。

### 6.2 NewAPI

#### 当前主要问题

1. 概览页虽然已有趋势图，但 `Token 消耗 / 分钟` 仍停留在总量表达，缺少用户真正关心的渠道维度趋势。
2. `渠道错误率前 10` 等排行型面板有价值，但主视图仍偏“运维/工程排障”，不够贴近“业务消耗变化”。
3. `模型请求速率前 10` 对运营或业务使用者不是第一优先级，应该让位于渠道维度 token 趋势这类更贴近问题的视图。
4. 标题虽已中文化，但整体仍较像指标描述，不够像问题表达。

#### 建议整改

- 增加按 `channel` 维度的 token 趋势图，放在高优先级位置。
- 保留渠道错误率排行，但将其定位为补充健康面板，而非最核心业务面板。
- 检查哪些总量 stat 应改为带趋势的小图或主趋势区，以减少“只有一个数”的阅读疲劳。

### 6.3 Agent Team

#### 当前主要问题

1. 当前看板已经有“事项创建 / 关闭趋势（小时 / 天）”，但仍偏 exporter 视角，未明确强化用户想看的“按项目维度新增/完成情况”。
2. `事项状态概览`、`尝试状态概览`、`角色积压分布`、`项目积压分布` 都是 bargauge，主页面分布面板偏多，阅读密度大。
3. `completion_mode`、`failure_code` 等开发/排障字段在次级视图有价值，但不应继续占据高优先级用户视角。
4. 用户当前更像是在问“哪个项目今天新增多、哪个项目完成多、节奏如何”，而不是“attempt status 有多少种分布”。

#### 建议整改

- 强化 `project` 维度的新增/完成趋势为核心面板。
- 将分布型概览从“主结论”降级为“补充分布”，减少首页 bargauge 密度。
- 保留 recovery / failure_code / completion_mode，但位置应更偏运维/排障，而不是总览主视图。

### 6.4 主机系统

#### 当前主要问题

1. 主机系统看板整体已经更偏人类可读，但 `Top 进程 CPU/内存` 仍是 bargauge。
2. 用户当前已明确偏好 timeseries，不过主机看板的 TopN 排名类问题仍有保留 bargauge 的合理性。

#### 建议整改

- 主机看板不应作为第二轮大动刀对象。
- 仅在视觉统一时做轻量风格整理，不建议再拆出独立 follow-up。

### 6.5 Arena / Uptime Kuma

#### 当前主要问题

1. 这两个系统当前用户反馈中不是最强诉求点。
2. 但同样应受第二轮人类可读性基线约束，避免继续残留明显难读或长期空置面板。

#### 建议整改

- 在 #33 通用整改中一并检查。
- 不单独作为本轮第一优先级新增指标对象，除非二次调研确认仍有关键缺口。

## 7. 第二轮缺失指标清单

### 7.1 OpenClaw

当前缺的不是“更多底层指标”，而是**更贴近用户语义的聚合视图**：

1. 按 `agent` 维度查看消息量变化趋势
2. 按 `agent` 维度查看 token 变化趋势
3. 若有必要，补一层 `agent` 标签映射或聚合逻辑，避免只能按 provider/model 看

### 7.2 NewAPI

当前缺的关键视角：

1. 按 `channel` 维度查看 token 变化趋势
2. 更直接反映“哪个渠道在消耗、波动、异常”的主视图

### 7.3 Agent Team

当前缺的关键视角：

1. 按 `project` 维度查看 issue 新增趋势
2. 按 `project` 维度查看 issue 完成/关闭趋势
3. 更贴近管理视角的节奏对比，而不是只看 backlog 分布

## 8. 第二轮标题与文案规范

第二轮主面板标题应尽量符合“人类问题语言”，而不是“指标字段语言”。

### 8.1 推荐写法

- `24h Token 总量` -> 可保留
- `按 Agent 看消息量变化` -> 优于 `Message Processed by Agent`
- `按渠道看 Token 变化` -> 优于 `Channel Token Consumption`
- `按项目看 Issue 新增趋势` -> 优于 `Issue Created Window Total by Project`

### 8.2 不推荐写法

- `Queue Wait 按 Lane`
- `Completion Mode 分布`
- `Channel / Outcome 异常分布`
- `Attempt Status Overview`

这些名称不是不能存在，而是：
- 不适合作为首页主视图标题
- 更适合作为排障页或 drill-down 页标题

## 10. 对实现侧的明确输入

后续实现 issue 至少应遵循以下规则：

1. **主视图优先回答用户问题，不优先展示底层维度。**
2. **趋势型问题优先用 timeseries，而不是只给排名或快照。**
3. **持续 No Data 的核心面板必须修复、替换或删除。**
4. **工程排障维度（lane、completion_mode、failure_code 等）保留在次级页面，不占首页核心位置。**
5. **标题、单位、图例、阈值要继续中文化且自然表达。**
6. **对于用户明确点名的三个增强方向，必须按独立验收目标建设，不得只靠现有总量面板“解释一下就算了”。**

## 11. PM 显式判断结论

本 issue 当前不需要继续拆分。

原因：
1. 当前 issue 已经是 PM 规格子 issue，本身就是独立验收单元。
2. 现有文档与 dashboard 产物已经足够支撑一份第二轮可读性与缺口清单。
3. 本文已经把第二轮目标拆成可独立验收的实现子项，继续拆分只会把规格本身切碎，增加编排噪音。
4. 当前最小正确动作是完成这份第二轮清单，然后让 #33-#36 继续各自实现，而不是在 PM 这里再制造更多 follow-up。

因此，`Issue #32` 在本文产出后即可视为完成，并建议直接关闭；随后由 `Issue #33` 到 `Issue #36` 按拆分结果继续推进。
